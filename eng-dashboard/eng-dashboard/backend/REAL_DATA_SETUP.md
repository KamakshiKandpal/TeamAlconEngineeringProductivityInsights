# Going from mock data to real GitLab data

## What changed
- `services/gitlabClient.js` ‚Äî real GitLab REST v4 client (auth, pagination, rate-limit backoff, retries)
- `services/normalizer.js` ‚Äî maps raw GitLab payloads into the same row shape `metrics.js`/`anomalies.js` already consume
- `db.js` ‚Äî Postgres upserts matching `db/schema.sql`
- `sync.js` ‚Äî the scheduled sync worker: fetch ‚Üí normalize ‚Üí upsert ‚Üí detect anomalies ‚Üí refresh rollups
- `.env.example` ‚Äî required config

`services/mockGitlab.js` is unchanged and still useful for local dev/demos without GitLab access ‚Äî nothing forces you to delete it.

## One-time setup
1. `cp .env.example .env` and fill in `GITLAB_BASE_URL`, `GITLAB_TOKEN`, `GITLAB_PROJECT_IDS`, `DATABASE_URL`.
2. Create the database and apply the schema:
   ```
   createdb eng_dashboard
   psql eng_dashboard -f db/schema.sql
   ```
3. `npm install` (pulls in `pg` and `dotenv` alongside the existing deps).

## Running a sync
```
npm run sync                    # syncs the last SYNC_WINDOW_HOURS (default 24h)
node sync.js --since=2026-06-01 # backfill from a specific date
```
Put this on a schedule ‚Äî cron, a k8s CronJob, GitHub Actions on a schedule trigger, or a BullMQ
repeatable job ‚Äî every 10‚Äì15 minutes. It's safe to overlap windows: every write is an upsert keyed
on the GitLab id, so re-processing the same MR or deployment twice is a no-op.

For true real-time updates (not just the 10‚Äì15 min sync cadence), add a GitLab webhook
(Project Settings ‚Üí Webhooks) pointed at a small receiver endpoint that calls the same
`normalizeMergeRequest`/`normalizeDeployment` + `db.upsert*` functions on each `merge_request` /
`deployment` event, instead of waiting for the next scheduled sync. `sync.js` then just becomes
the reconciliation pass that catches anything a missed/failed webhook didn't.

## Switching the API server off mock data
`server.js` currently reads from the in-memory arrays generated by `services/mockGitlab.js`
(`ALL_MRS`, `ALL_DEPLOYS`) so the whole stack runs standalone with zero setup. To point it at real
data, replace those in-memory reads with SQL queries through `db.js`'s `pool`, e.g.:

```js
// before (mock):
const mrs = applyFilters(MRS.filter(m => m.state === 'merged'), req.query, 'merged_at');

// after (real):
const { rows: mrs } = await db.pool.query(
  `SELECT * FROM merge_requests
   WHERE state = 'merged'
     AND ($1::int IS NULL OR team_id = $1)
     AND ($2::timestamptz IS NULL OR merged_at >= $2)
     AND ($3::timestamptz IS NULL OR merged_at <= $3)`,
  [req.query.team || null, req.query.from || null, req.query.to || null]
);
```

For the trend/KPI endpoints, query `mv_weekly_team_metrics` / `mv_weekly_deploy_freq` instead of
raw tables where possible ‚Äî that's the whole point of the materialized views in `db/schema.sql`:
sub-millisecond dashboard queries even with millions of historical MRs.

## Getting a GitLab token
Project/Group ‚Üí Settings ‚Üí Access Tokens ‚Üí create one with the **`read_api`** scope (read-only is
sufficient; this pipeline never writes to GitLab). For an org-wide dashboard, a **group access
token** on the top-level group is usually easiest since it covers every project underneath it.
